# よりそいPHRシリーズ 全体オーバービュー

- **作成**: 2026-05-16
- **状態**: ブレスト Phase 1 完了 → コンセプト言語化中
- **ブレスト元**: `.plans/designs/2026-05-16-yorisoi-phr-redesign-brainstorm.md`

## 1. PHRシリーズの親フレーズ

> **困った時、ここに帰ってこられる。**

「毎日使ってもらう」プレッシャーをかけない。寛解期患者の罪悪感をゼロにする。
平時は遠ざかってもいい。困った時に戻れる場所として PHRシリーズを位置づける。

## 2. シリーズ構成（3サブブランド + 入口AI）

| サブブランド | 役割 | 患者の場面 | 既存資産との対応 |
|---|---|---|---|
| **よりそい きろく** | 記録の核 | 急に体調が崩れた時の思い出し | 系統B（LINE音声整形）取り込み・既存「きろくする」発展 |
| **よりそい そうだん** | 相談の受け皿 | 人に話せない不安の受け皿 | うちあけRAG×AIチャット・添田白血病版+talk統合 |
| **よりそい てちょう** | 医療パスポート | 診察前の「あれ何だっけ」に備える | 既存「visits/clinics」「みせる」発展 |

### 入口AI（カウンセラー）

サブブランド名を最初に見せない優しい入口を作る。
カウンセラーAI（既存talkのカウンセラー化版）が「今日はどんな感じですか」と聞き、患者の言葉から状態を判定して、適切なサブブランドへそっと誘導する。

これは5/11布谷さん発案の「**何で困ってますか？**」フローを発展させたもの。布谷さん指摘の「**チャンスの女神は前髪しかない**」を入口AIで掴む。

## 3. 状態適応（案ε のもう一つの軸）

各サブブランドは、患者の状態を判定してUIを変化させる：

| 状態 | きろく | そうだん | てちょう |
|---|---|---|---|
| **急性期/しんどい** | 最小記録（体重・体温・血便・メモ） | カウンセラー型（共感のみ） | 緊急カード（あんしんカード） |
| **安定/寛解期** | 詳細記録（音声・写真・タグ） | RAG提案型（経験者の声） | 多科併診サマリー |
| **新規発症期** | テンプレ誘導型（書き方ガイド） | 診断理解モード | 病院・科を登録 |
| **発信期** | 振り返り→うちあけ投稿導線 | 自分の経験を整理 | 体験談アーカイブ |

→ 状態判定は入口AIで行うか、もしくは各サブブランド内で軽く問う方式。

## 4. ターゲット患者

**コア**: 重い/進行性/希少疾患の患者、患者会所属者
- 白血病・IBD中重度・パーキンソン・SLE・難病結合組織病・POEMS等
- 5/14内部ブレスト合意「ノートをつけている人」層

**取り込み余地**: 主治医以外の相談相手が欲しいすべての患者（軽症含む）

## 5. 「よりそい」全体地図（PHRシリーズの位置）

「よりそい」はプロダクトファミリーで、PHRシリーズはその一部：

```
よりそい（親ブランド）
├─ よりそい 光（hikari）       ← 医療機関向け・本番稼働
├─ よりそい（音声）             ← LINE音声→整形・本番稼働（系統B）
├─ よりそい きろく              ← PHRシリーズ・記録（系統Cの一部・新）
├─ よりそい そうだん            ← PHRシリーズ・相談（系統Cの一部・新）
├─ よりそい てちょう            ← PHRシリーズ・医療パスポート（系統Cの一部・新）
└─ よりそい つながる            ← 患者SNS・別リポジトリ（系統D）
```

PHRシリーズ内の3サブブランドは**共通の患者プロファイル基盤**を見るが、入口・体験は分かれている。
「つながる」はデータも完全独立。「よりそい（音声）」は API 連携で「きろく」にデータ流入。

## 6. データフロー（重要）

```
[患者の診察]
    ↓ LINE音声送信
[よりそい（音声）= 系統B]
    ↓ Gemini整形 → GCS保存
    ↓ /summary/:sessionId API
[よりそい きろく]
    ↓ 整形済み記録が自動でタイムラインに乗る
[よりそい てちょう]
    ↓ 多科併診サマリー生成
    
[患者の不安]
    ↓ AIチャット
[よりそい そうだん]
    ↓ うちあけ API ← 経験者の声を引く（RAG）
    ↓ 自分の記録（きろく/てちょう）も参照
    ↓ 主治医への相談下書きを生成
```

→ 3サブブランドは独立した入口を持つが、データは互いに参照する。
→ 系統Bは「きろく」のデータ源として位置づける（廃止しない・別物として共存）。
→ うちあけは「そうだん」のRAG源として位置づける（データ蓄積段階に応じて疾患を段階展開）。

## 7. 6/2大阪IBD再提示でどう見せるか（叩き）

優先順位の提案：

1. **「よりそい きろく」のデモ** — 5/11フィードバック反映済みの最小記録モード+食事写真+カウンセラーAI化を披露
2. **「よりそい そうだん」の構想** — うちあけRAGで「同じIBD患者の声から学ぶ」体験。IBDはデータ蓄積が比較的進めやすい
3. **「よりそい てちょう」の現状** — 既存のみせる/ふりかえるをリブランド

→ 6/2は「シリーズ化したことを伝える」が主目的。完璧な実装は不要。
→ 商談で「これ欲しい」を引き出すたたきとして、サブブランドの**コンセプトカード**を3枚用意するのが現実解。
